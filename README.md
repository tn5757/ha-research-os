# Research OS - 設計思想

## このプロジェクトは何か

**研究マネジメントOS** ― 一人の研究者が、一貫性のある研究プログラムをもって論文を生み出すための基盤。

---

## 核心となる問題意識

「良い論文を一本書く」こと自体は、やろうと思えばできる。

しかし、論文を個別に書き始めてしまうと、研究テーマ・理論的枠組み・方法論・学会発表が全部バラバラになって、**「結局この研究者は何を明らかにしたい人なのか」が見えなくなる**。

だから必要なのは、個々の論文を作り込むことよりも、**それらが"同じ問いの系譜の上にある状態"を先につくること**。

---

## 設計原則

### 1. 問いが先、論文は後

```
「何を書くか」より「何を問うているのか」を先に決める
```

具体的には：
- この研究プログラム全体を貫く核心的な問いは何か
- その問いは、現場のどんな「未解決の困難」に根ざしているか
- その問いに対して、既存研究はどこまで答えていて、どこが空白か
- その空白に対して、自分だからこそ取れるアプローチは何か

**これを先に固定しておく。**

この研究プログラムレベルの前提さえ揃えば、個々の論文は"問いの体系を埋める作業"になる。一本一本が完璧でなくても、全体が一つの知的営みとして積み上がる。学会発表も投稿論文も、同じ問題意識から語れる。

### 2. 知は差分で積み上がる

研究の進展とは「新しいことを知る」ことではない。**「すでにわかっていること」に対して「新しい素材」がどのような差分をもたらすかを特定すること**。

```
02_insights/concepts/  ← これまでにわかっていること（既存知見の蓄積）
        ↑ 照合
01_insight-sources/    ← 新しい素材（論文、観察、議論）
        ↓
    差分の特定           補強？ 修正？ 対立？ 拡張？ 新規？
        ↓
    研究のヒント         「だとすると、次に問うべきはこれだ」
        ↓
02_insights/research-questions/  ← 新しいRQ
        ↓
    新しい探索           RQに答えるための文献・観察・議論
        ↓
01_insight-sources/    ← 新しい素材が増える
        :
      （螺旋が続く）
```

この螺旋の要は `02_insights/concepts/` が果たす二重の役割にある：

- **出力先**: 新しいinsightが書き込まれる場所
- **照合先**: 次のinsightを生成するときの比較基盤

conceptsが蓄積されるほど、次のinsightの差分はより鮮明になり、研究のヒントはより具体的になり、RQはより鋭くなる。**知の蓄積が次の知の質を規定する。**

### 3. 上位概念が下位概念を規定する

螺旋で知を積み上げつつも、全体の方向は `canon` が規定する。

```
00_canon/            ← 研究者としての核心的問い・パラダイム
  ↓ 方向づけ
01_insight-sources/  ← 何を読み、何を観察し、何を議論したか
  ↓ 蒸留（既存conceptsとの差分）
02_insights/         ← リサーチクエスチョン・概念・研究のヒント
  ↓ 構成
03_papers/           ← 論文（セクション単位で管理）
```

Claudeが下位のドキュメントを生成するとき、必ず上位を参照する。だから何を書いても一貫する。

### 4. 現場の問いから出発する

研究も「誰のための知か」という視点で捉える。

**研究課題の3条件：**
1. **困難が"現在進行形"** ― 理論上の興味ではなく、現場で「今、困っている」
2. **構造的要因が見える** ― 「仕組みや前提がズレている」と言語化できる
3. **知の空白がある** ― 既存研究が「扱っている風」でも、本質に届いていない

**研究としての成立性チェック：**
1. **学術的貢献は明確か** ― 先行研究の空白を埋め、理論的・実践的に前進するか
2. **実行可能か** ― データへのアクセス、倫理的制約、時間的制約の中で遂行できるか
3. **波及するか** ― 一本の研究が次の問いを生み、プログラム全体を前に進めるか

### 5. ポジショニングは研究ギャップの選定

ポジショニングは「新しいテーマを見つけること」ではない。**「どの空白で勝負するか」の選択**。

- 先行研究がどの領域をカバーしているか
- まだ十分に探究されていない空白はどこか
- その空白に対して、自分の強み（専門性・フィールド・方法論）で切り込めるか

**空白を選んでから、リサーチクエスチョンが定まる。**

### 6. 一貫性 > 完璧性

すべての論文が完璧に仕上がっている必要はない。

重要なのは、**どの論文を見ても「この研究者は何を明らかにしようとしているか」が伝わること**。

## プロジェクト構成

```
research-os/
│
├── 00_canon/              研究者としての核心的問い・パラダイム
│                          「自分は何を明らかにしたい人か」を宣言する
│
├── 01_insight-sources/    insightsを生むすべてのインプット（思考の素材）
│   ├── literature/            先行研究（論文・書籍の要約とメモ）
│   ├── field-notes/           フィールドノート（授業観察等）
│   ├── meeting-logs/          MTGログ（ブレスト、研究会議の記録）
│   └── web-research/          /web-research の実行結果（日時付きログ）
│
├── 02_insights/           蒸留されたリサーチクエスチョン・概念・研究のヒント
│   ├── concepts/              これまでにわかっていること（照合先 兼 出力先）
│   └── research-questions/    問いの一覧と発展の記録
│
├── 03_papers/             論文（セクション単位で管理）
│   ├── references/            引用文献プール（全論文で共有する書誌情報）
│   └── paper-slug/
│       ├── _meta.md           タイトル・投稿先・ステータス・日付・DOI
│       ├── citations.md       この論文の引用一覧（何を・どこを引用したか）
│       ├── data/
│       │   ├── raw/           生データ（アンケート回答、逐語録等）
│       │   └── processed/     加工済みデータ（コーディング結果、統計処理後）
│       ├── 01_introduction.md
│       ├── 02_literature-review.md
│       ├── 03_methodology.md
│       ├── 04_findings.md     ← data/ を直接参照して執筆
│       ├── 05_discussion.md
│       └── 06_conclusion.md
│
└── lib/                   変わりにくい参照知識
    ├── submission-guidelines/  投稿規定（ジャーナルごと）
    ├── domain-knowledge/       ドメイン知識
    └── methodology/            方法論テンプレート
```

### 論文のライフサイクル

論文はフォルダを移動しない。`_meta.md` の `status` だけが進む。

```
drafting → submitted → in_review → revision → accepted → published
```

投稿済み・出版済みの論文も同じ場所に残る。次の論文のDiscussionで自分の過去論文を引用するなど、**研究プログラム内での参照が続く**ため。変更の経緯はgit履歴が保持する。

### 各層の関係

| 層 | 役割 | Claudeの参照パターン |
|---|---|---|
| `00_canon/` | 方向の錨 | すべての生成で参照 |
| `01_insight-sources/` | 思考の素材 | insight生成・文献レビュー・考察の生成時 |
| `02_insights/concepts/` | 既存知見の蓄積（照合先 兼 出力先） | insight生成時に照合、RQ策定・論文構想時に参照 |
| `02_insights/research-questions/` | 問いの体系 | 論文構想・新RQ生成時に重複確認 |
| `03_papers/references/` | 引用の台帳（共有） | 引用挿入時 |
| `03_papers/*/citations.md` | 論文別の引用記録 | 各セクション執筆時 |
| `03_papers/*/data/` | 一次データ（証拠） | Findings・考察の執筆時 |
| `03_papers/` | 最終成果物 | セクション単位で執筆・推敲 |
| `lib/` | 制約と型 | 投稿規定の遵守・方法論の適用 |

## なぜClaude Codeか

フォルダの中に知識を構造的に正しい形で置いておけば、Claudeが必要な情報をほぼ自動で読み取って、成果物を作ってくれる。

- `00_canon/` に問いを置けば、すべての生成物がその問いに沿う
- `01_insight-sources/` にメモを蓄積すれば、文献レビューの素材になる
- `02_insights/concepts/` に知見が蓄積されるほど、次のinsightの差分が鮮明になる
- `03_papers/references/` に書誌情報があれば、正確な引用を挿入できる
- `03_papers/*/data/` に一次データを置けば、Findingsの根拠として直接参照できる
- `03_papers/` の `_meta.md` に投稿先を書けば、`lib/` の投稿規定を自動参照する

**人間は「問い」を立て、「読み」、「考える」。Claudeは「一貫性を保ちながら構成する」。**

## この思想が実現すること

```
従来の問題:
  論文A ──→ 学会発表
  論文B ──→ 紀要投稿     → バラバラ、蓄積しない
  論文C ──→ 共同研究

Research OSの世界:

  01_insight-sources/  ──→  02_insights/concepts/  ──→  03_papers/
        ↑                     ↕（照合 × 蓄積）            │
        │                  research-questions/             │
        │                     │                            │
        └──── 研究のヒントが ──┘                            │
              新しい探索を駆動                               ↓
                                                  すべて 00_canon/ に貫かれた
                                                  一貫した研究プログラム
```

---

*「各論より先に問いを固め、差分で知を積み上げる」― これがResearch OSの設計思想である。*
